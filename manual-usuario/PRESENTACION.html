🏗️ Arquitectura Cliente-Servidor

Componentes

CLIENTE (Angular :4200)
  │  AuthInterceptor: adjunta JWT
  │  AuthGuard: protege rutas por rol
  │  HttpClient → API REST
  ▼
SERVIDOR (Express :3000)
  │  Helmet → CORS → Rate Limit
  │  authMiddleware(roles)
  │  Rutas: /materiales, /movimientos
  │         /api/login, /api/usuarios
  ▼
SQLite (database.sqlite)
  Tablas: materiales, movimientos, usuarios

Flujo de Autenticacion

Login → POST /api/login
  → bcrypt.compare(password, hash)
  → jwt.sign({id, rol})
  → {token, user}

Cada request:
  → AuthInterceptor agrega header
  → Authorization: Bearer <token>
  → authMiddleware.verify(token)
  → Si valido → next()
  → Si invalido → 401

Token expira en 12h
🔐 JWT en localStorage (ckj_token) 🛡️ 3 roles: admin, almacen, ventas ⏱️ Rate limit: 100 req/15min + 5 login/15min

Codigo: Proteccion de rutas con AuthGuard

export class AuthGuard implements CanActivate {
  canActivate(route: ActivatedRouteSnapshot): boolean {
    if (!this.auth.isLoggedIn()) {                  // 1. Esta logueado?
      this.router.navigate(['/login']); return false; }
    const roles = route.data['roles'] as string[];
    if (roles && !this.auth.hasRole(roles)) {       // 2. Tiene el rol?
      this.router.navigate(['/dashboard']); return false; }
    return true;                                   // 3. Acceso permitido
  }
}

🗄️ Base de Datos SQLite - Esquema Completo

Tabla: materiales

CREATE TABLE materiales (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  nombre TEXT NOT NULL,
  descripcion TEXT DEFAULT '',
  categoria TEXT DEFAULT '',
  cantidad REAL NOT NULL DEFAULT 0,
  unidad TEXT DEFAULT 'unidades',
  ubicacion TEXT DEFAULT '',
  fechaIngreso TEXT DEFAULT (date('now')),
  proveedor TEXT DEFAULT '',
  precioUnitario REAL DEFAULT 0,
  stockMinimo REAL DEFAULT 10
);

Tabla: movimientos

CREATE TABLE movimientos (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  materialId INTEGER NOT NULL,
  tipo TEXT NOT NULL CHECK(tipo IN ('ingreso','salida')),
  cantidad REAL NOT NULL,
  fecha TEXT DEFAULT (datetime('now','localtime')),
  proveedor TEXT DEFAULT '',
  destino TEXT DEFAULT '',
  usuario TEXT DEFAULT 'Admin',
  comprobante TEXT DEFAULT '',
  FOREIGN KEY (materialId) REFERENCES materiales(id)
);

Tabla: usuarios

CREATE TABLE usuarios (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT UNIQUE NOT NULL,
  password TEXT NOT NULL,     -- almacenado con bcrypt (10 rondas)
  nombre TEXT NOT NULL,
  rol TEXT NOT NULL DEFAULT 'almacen',  -- 'admin' | 'almacen' | 'ventas'
  activo INTEGER DEFAULT 1
);

📋 CRUD de Materiales - Codigo Clave

Los 7 endpoints del inventario

// GET - Listar todos (publico)
app.get('/materiales', (req, res) =>
  res.json(queryAll('SELECT * FROM materiales ORDER BY id DESC')));

// POST - Crear (solo admin/almacen)
app.post('/materiales', authMiddleware(['admin','almacen']), (req, res) => {
  const { nombre, descripcion, categoria, cantidad, unidad,
            ubicacion, proveedor, precioUnitario, stockMinimo } = req.body;
  runSql('INSERT INTO materiales VALUES (NULL,?,?,?,?,?,?,?,?,?,?)',
    [nombre, descripcion||'', categoria||'', cantidad||0, unidad||'unidades',
     ubicacion||'', new Date().toISOString().split('T')[0],
     proveedor||'', precioUnitario||0, stockMinimo ?? 10]);
  res.status(201).json(queryOne('SELECT * FROM materiales WHERE id=?', [getLastId()]));
});

// POST - Ajustar stock (solo admin, crea movimiento automatico)
app.post('/materiales/:id/ajustar', authMiddleware(['admin']), (req, res) => {
  // Actualiza cantidad y registra movimiento con motivo
  runSql('UPDATE materiales SET cantidad=? WHERE id=?', [cantidad, req.params.id]);
});
🔒 Queries parametrizadas: previenen SQL Injection🔑 authMiddleware: control de acceso por rol

Capa 1: Backend

Helmet (headers HTTP seguros)
express-rate-limit (100 req/15min)
Login: 5 intentos/15min
express.json({limit:'1mb'}) anti-DoS

Capa 2: Autenticacion

JWT con expiracion 12h
bcryptjs 10 rondas de salt
authMiddleware por rol
JWT_SECRET desde variable de entorno

🔍

Capa 3: DevSecOps

npm audit en CI/CD y pre-commit
Gitleaks: detecta secretos en codigo
Trivy: escanea vulnerabilidades
SonarQube: calidad de codigo

🌐 Nginx - Configuracion del Reverse Proxy

server {
    listen 443 ssl http2;
    server_name ckj.local;                    # Cambiar por dominio real

    ssl_certificate     /etc/letsencrypt/live/dominio/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/dominio/privkey.pem;

    root /var/www/ckj/dist/ckj/browser;       # Archivos estaticos Angular
    index index.html;

    # Frontend SPA - todas las rutas a index.html
    location / { try_files $uri $uri/ /index.html; expires 1y; }

    # API Backend - proxy a Express en puerto 3000
    location /api/ { proxy_pass http://127.0.0.1:3000; }

    # Endpoints del backend sin prefijo /api/
    location ~ ^/(materiales|movimientos|dashboard) {
        proxy_pass http://127.0.0.1:3000;
    }

    # Archivos subidos (comprobantes)
    location /uploads/ { alias /var/www/ckj/backend/uploads/; expires 30d; }

    location ~ /\. { deny all; }              # Seguridad: ocultar archivos . (dotfiles)
}

# .github/workflows/ci-cd.yml - Los 3 jobs
jobs:
  frontend:
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build -- --configuration=production && npm test

  backend:
    steps:
      - run: cd backend && npm ci && node --check server.js

  devsecops:
    steps:
      - run: npm audit --audit-level=high          # Escanea dependencias
      - run: grep -rn "password\|secret" src/ backend/ || true  # Busca secretos

Flujo del commit

       ↓
  Paso 1: Prettier formatea codigo
          (archivos .ts, .js, .json,

       ↓
  Paso 2: Gitleaks busca secretos
          (passwords, API keys, tokens)

  ❌ Error → commit rechazado

📦 Docker - Build Multi-stage

# Dockerfile con 3 etapas: build frontend → build backend → final
# Imagen final: solo lo necesario, usuario no-root

# Etapa 1: Compilar Angular
FROM node:22-alpine AS build-frontend
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build -- --configuration production

# Etapa 2: Preparar backend
FROM node:22-alpine AS build-backend
WORKDIR /app
COPY backend/package*.json ./backend/
RUN cd backend && npm ci
COPY backend/ ./backend/

# Etapa 3: Imagen final (usuario no-root seguridad)
FROM node:22-alpine
RUN addgroup -S ckj && adduser -S ckj -G ckj
COPY --from=build-frontend --chown=ckj:ckj /app/dist ./dist
COPY --from=build-backend --chown=ckj:ckj /app/backend ./backend
USER ckj                                  # No ejecutar como root
CMD ["node", "backend/server.js"]
